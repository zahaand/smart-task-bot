# Implementation Plan: Button-Driven UX

**Branch**: `003-button-driven-ux` | **Date**: 2026-03-19 | **Spec**: [spec.md](spec.md)

---

## Summary

Add a full button-driven interaction layer to the Smart Task Bot so that all core flows (create task, view tasks, set reminder, mark done, delete) can be completed without typing commands. The primary deliverables are: a persistent reply keyboard, a per-user state machine (`UserState` + `UserStateService`), an inline calendar for date selection, a free-text time parser, and inline action buttons on every task. All existing text commands remain fully functional.

---

**Scale/Scope**: Single-user bot for demo; no concurrency concerns beyond existing design.

## Architecture Deep-Dive

### State Machine — UpdateDispatcher Routing

```
onUpdateReceived(update)
  └─► dispatcher.dispatch(update)
        ├─ hasCallbackQuery? → routeCallback(update)
        │     startsWith("tz:")             → TimezoneCallbackHandler
        │     startsWith("TASK_REMIND:")    → TaskActionCallbackHandler
        │     startsWith("TASK_DONE:")      → TaskActionCallbackHandler
        │     startsWith("TASK_DELETE:")    → TaskActionCallbackHandler
        │     startsWith("CAL_DATE:")       → CalendarCallbackHandler
        │     startsWith("CAL_NAV:")        → CalendarCallbackHandler
        │     startsWith("CONFIRM_DELETE:") → DeleteConfirmCallbackHandler
        │     equals("CONFIRM_CANCEL")      → DeleteConfirmCallbackHandler
        │     startsWith("TASKS_TAB:")      → TaskListTabCallbackHandler
        │
        └─ hasMessage + hasText? → routeMessage(update)
              1. userStateService.resetIfStale(userId)
              2. state = userStateService.getState(userId)
              3. isCommand(text)?  [text.startsWith("/")]
                 └─ text == "/cancel" AND state != IDLE
                    → userStateService.cancelWithNotification(userId, chatId, state) → return
                 └─ else → command switch (step 7) — commands always bypass state handlers
              4. isPersistentMenuButton(text)?
                 └─ state != IDLE → userStateService.cancelWithNotification(userId, chatId, state)
                 └─ route to button handler (below)
              5. state == CREATING_TASK?          → TaskCreationTextHandler
              6. state == ENTERING_REMINDER_TIME? → ReminderTimeTextHandler
              7. state == CONFIRMING_DELETE
                 OR state == SELECTING_REMINDER_DATE?
                 → notificationService.sendMessage(chatId, "Пожалуйста, используй кнопки выше.")
                 → return (stay in current state)
              8. BTN_NEW_TASK → registrationGuard → NewTaskButtonHandler
                 BTN_MY_TASKS or BTN_REMINDER → registrationGuard → TaskListButtonHandler
              9. command switch (unchanged, backward compat)
                 /start /help /newtask /tasks /remind /done
             10. default → UnknownInputHandler
```

**Routing rules (summary)**:
- `/cancel` always interrupts any active flow regardless of state — checked first (step 3) so it can never be swallowed as task text.
- Any other command always bypasses state handlers and reaches the command switch (step 9) — commands are never treated as input text.
- `CONFIRMING_DELETE` and `SELECTING_REMINDER_DATE` await inline button taps only; free-text input in these states returns `"Пожалуйста, используй кнопки выше."` without changing state (step 7).
- Persistent menu buttons interrupt non-IDLE text-input states (CREATING_TASK, ENTERING_REMINDER_TIME) by cancelling with notification and starting the new flow (step 4).

### UserStateService Responsibilities

```
getState(Long userId)                    → ConversationState (IDLE if no row)
setState(Long userId, ConversationState) → upsert row, clear context, set updated_at
setStateWithContext(Long userId,
  ConversationState, ConversationContext) → upsert row with serialized context
getContext(Long userId)                  → Optional<ConversationContext>
updateContext(Long userId,
  ConversationContext)                   → update context field only, touch updated_at
resetIfStale(Long userId)                → if updated_at > 24h ago → setState(IDLE)
cancelWithNotification(Long userId,
  Long chatId, ConversationState active) → setState(IDLE) + notificationService.sendMessage(
                                           chatId, cancelMessage(active))
```

`cancelMessage(state)` returns a human-readable string:
- CREATING_TASK → `"Создание задачи отменено."`
- ENTERING_REMINDER_TIME → `"Установка напоминания отменена."`
- CONFIRMING_DELETE → `"Удаление отменено."`
- SELECTING_REMINDER_DATE → `"Выбор даты отменён."`

### CalendarKeyboardBuilder — Calendar Layout

```
buildCalendar(int year, int month, Long taskId) → InlineKeyboardMarkup

Row 0: "« Апрель 2026 »" (header, non-clickable — callback "NO_OP")
Row 1: "Пн  Вт  Ср  Чт  Пт  Сб  Вс"  (non-clickable)
Rows 2-N: date buttons
  - Past dates (before today): button text "·", callback "NO_OP"
  - Today and future: button text "15", callback "CAL_DATE:2026-04-15"
  - Dates outside this month: button text " ", callback "NO_OP"
Navigation row:
  - Previous month button: "←" → callback "CAL_NAV:-1" (disabled if current month)
  - Next month button: "→"   → callback "CAL_NAV:+1"
```

For the disabled prev-month button when showing the current month: still render the "←" button but with callback `"NO_OP"` so it does nothing (rather than omitting — keeps layout symmetric). Alternatively, show "‹" in a dimmed style — text only, same `NO_OP` callback.

`NO_OP` should be defined as a `BotConstants.CB_NO_OP = "NO_OP"` constant. `UpdateDispatcher.routeCallback()` must answer the callback query silently when data equals `NO_OP`.

### TaskListKeyboardBuilder — Task Row Layout

Each active task:
```
[⏰ Напомнить]  [✅ Выполнить]  [🗑 Удалить]
```
Each completed task:
```
[🗑 Удалить]
```
Tab row (always present, appended at bottom):
```
[Активные ✓]  [Выполненные]   (when showing active tab)
[Активные]    [Выполненные ✓]  (when showing completed tab)
```
Tab buttons: callbacks `TASKS_TAB:ACTIVE` and `TASKS_TAB:COMPLETED`.

30. **`NotificationService.editMessage` fallback** — all `editMessageText` calls (task list tab toggle, calendar navigation) MUST be wrapped in try-catch for `TelegramApiException`. This applies to: `editTaskList`, `editCalendar`, and any future `editMessage` calls. Implement as a private `safeEdit(EditMessageText, SendMessage fallback)` helper in `NotificationService`.

## Key Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Stale calendar message (user closes chat, returns days later, taps old "←") | `resetIfStale` in `routeCallback` resets state before processing; handler checks state before acting; if state no longer `SELECTING_REMINDER_DATE`, handler sends "Сессия истекла, начни заново" and returns IDLE |
| `editMessageText` fails (message too old, >48h Telegram limit) | Wrap in try-catch; on failure, send new message instead; log WARN |
| JSONB context deserialization failure (malformed JSON) | `UserStateService.getContext()` catches `JsonProcessingException`, resets state to IDLE, logs ERROR |
| Task deleted between CONFIRMING_DELETE state set and confirm tap | `DeleteConfirmCallbackHandler` handles `0 rows deleted` from repository → sends "Задача уже была удалена" and returns IDLE |
| 64-byte callback data limit breach | All prefixes verified in contract; `taskId` is a `BIGINT` → max 19 digits = longest realistic callback is `CONFIRM_DELETE:9999999999999999999` = 36 chars |
